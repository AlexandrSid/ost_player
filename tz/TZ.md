## Цель

Сделать локальный портативный (portable) плеер OST/музыки из папок игр, ориентированный на Windows 11, с управлением **глобальными горячими клавишами** и минимальным интерфейсом **в терминале (TUI/CLUI)**. Плеер должен уметь индексировать треки из набора папок, фильтровать “мусорные”/мелкие аудиофайлы по порогу размера, поддерживать shuffle/repeat и работу с “плейлистами” как наборами папок.

## Целевая платформа

- **Основная**: Windows 11 (обязательное корректное поведение).
- **Дополнительно (best effort)**: Linux Mint. Возможны ограничения в части “глобальных хоткеев” в зависимости от X11/Wayland; для Windows функционал обязателен.

## Определения

- **Источник (source folder)**: папка на диске, внутри которой рекурсивно ищутся треки для воспроизведения.
- **Трек**: файл в поддерживаемом аудиоформате (минимум: `.mp3`, `.ogg`) удовлетворяющий фильтрам (например, размер >= порога).
- **Плейлист**: именованный блок в конфигурационном файле, содержащий **перечень папок-источников** для воспроизведения. Плейлист не является списком конкретных файлов; файлы вычисляются индексатором на основе папок.
- **Активный плейлист**: плейлист, выбранный пользователем в меню; именно его папки используются для построения очереди треков.

## Ключевые пользовательские сценарии (workflow)

### Запуск и выбор источников

1) Пользователь запускает приложение из терминала (PowerShell/CMD/Windows Terminal).
2) Приложение показывает меню:
   - список плейлистов (или активный плейлист + список папок активного плейлиста)
   - пункты:
     - `1` добавить папку
     - `2` удалить папку(и)
     - `3` воспроизвести выбранное
     - `4` настройки (size threshold, shuffle mode, repeat mode)
     - `5` плейлисты (создать/переименовать/удалить/переключить)
3) Пока пользователь не подтвердил воспроизведение — приложение циклично находится в этом меню, позволяя добавлять/удалять папки и управлять плейлистами.

### Воспроизведение в фоне

1) Пользователь выбирает `3` (воспроизвести).
2) Запускается воспроизведение очереди треков.
3) Пользователь сворачивает терминал — воспроизведение **продолжается**.
4) Управление происходит через **глобальные** хоткеи (не требующие фокуса на терминале).

## Требования к портативности (Portable)

- Приложение поставляется как папка, которую можно разместить в **произвольном месте** (корень диска, пользовательская папка и т.п.).
- Приложение **не должно**:
  - требовать инсталлятора,
  - записывать данные в `%APPDATA%`, реестр, “ProgramData”, системные каталоги,
  - ставить сервисы/демоны/драйверы,
  - менять системные настройки.
- Все данные приложения (конфиг, плейлисты, кеш) должны храниться **внутри папки приложения**.
- Если папка приложения **не доступна на запись** (например, приложение положили в `C:\\Program Files\\...`), приложение должно:
  - выдать понятное сообщение об ошибке,
  - подсказать перенести папку приложения в директорию с правами записи (например `D:\\Apps\\ost_player\\`),
  - завершиться без повреждения данных.

## Файловая структура приложения

Минимальная структура (обязательная):

- `ost_player.exe` (или другое имя бинарника)
- `data/`
  - `config.toml` (или `config.json`) — основной конфиг
  - `playlists/` — опционально, если плейлисты хранятся отдельными файлами (см. ниже)
  - `cache/` — опционально
  - `logs/` — опционально

Разрешено упрощение:
- хранить плейлисты **внутри одного файла** `data/config.*` блоками (предпочтительно по требованию “плейлист — блок в файле сохранённой конфигурации”).

## Конфигурация и плейлисты

### Общие требования

- Конфиг должен быть:
  - человеко-редактируемым (ручное редактирование файла),
  - устойчивым к ошибкам (валидация; при ошибке — понятное сообщение и безопасный отказ от запуска/применения).
- Конфиг должен поддерживать:
  - глобальные настройки проигрывания,
  - список плейлистов как именованных блоков,
  - хоткеи (настраиваемые).

### Модель данных (логическая)

- `settings`:
  - `min_size_bytes` — порог минимального размера файла
  - `shuffle_mode` — режим shuffle (минимум: on/off)
  - `repeat_mode` — минимум: off / repeat_one / repeat_all
  - `supported_extensions` — список расширений (по умолчанию: mp3, ogg)
- `playlists`:
  - `name` — имя плейлиста
  - `folders[]` — список путей к папкам-источникам (абсолютные пути)
- `active_playlist` — имя/идентификатор активного плейлиста
- `hotkeys`:
  - биндинги команд на комбинации
  - параметры tap/hold (тайминги)

### Требование к плейлистам (обязательное)

- Плейлист — **блок в файле сохранённой конфигурации**, включающий:
  - название
  - перечень папок
- В TUI должен быть отдельный пункт меню для управления плейлистами:
  - создать плейлист
  - удалить плейлист
  - переименовать плейлист
  - переключить активный плейлист (swap)
  - редактировать список папок активного плейлиста (через пункты 1/2 меню или внутри подменю плейлистов — на усмотрение реализации)

## Индексация и фильтрация треков

### Поиск файлов

- Поиск треков выполняется **рекурсивно** по всем папкам активного плейлиста.
- Дедупликация:
  - одинаковые пути не должны появляться несколько раз,
  - одинаковые папки-источники должны игнорироваться при повторном добавлении.

### Фильтры

- **Фильтр по расширению**: по умолчанию добавлять только `.mp3`, `.ogg`. (Расширяемо конфигом.)
- **Фильтр по размеру**:
  - не добавлять в плейлист/очередь файлы, чей размер меньше `min_size_bytes`.
  - `min_size_bytes` должен настраиваться в TUI (пункт “настройки”) и/или через конфиг.

### Обработка ошибок

- Если папка-источник не существует/недоступна:
  - не падать с паникой,
  - показать сообщение в TUI/логах,
  - продолжить с остальными папками (если возможно).
- Если файл не удается декодировать:
  - пропустить файл,
  - перейти к следующему треку,
  - отметить ошибку пользователю (коротко) и/или в лог.

## Воспроизведение

### Форматы

- Минимум: mp3, ogg.
- Дополнительно: по желанию расширяемо.

### Очередь и порядок

- Базовый порядок: порядок, соответствующий результату индексации (уточнить в реализации: например, сортировка по пути/имени).
- Shuffle:
  - при включении shuffle — перемешать очередь, сохранив текущий трек (если уже играет).
- Repeat:
  - `repeat_off`: по окончании очереди остановить воспроизведение
  - `repeat_one`: повторять текущий трек
  - `repeat_all`: повторять всю очередь

### Seek / Fast-forward / Rewind

- При удержании хоткея “вправо/влево” выполнять fast-forward/rewind.
- Дефолтный режим (обязательный): **ступеньками по 5 секунд** с повтором каждые 200–300ms, пока клавиша удерживается.

## Горячие клавиши (глобальные)

### Общие требования

- Хоткеи должны работать **глобально**, когда терминал свернут и не в фокусе.
- Хоткеи должны быть настраиваемыми через конфиг.
- Должны быть дефолтные хоткеи.

### Дефолтные биндинги (Windows 11)

Комбинация-модификаторы по умолчанию: `Ctrl + RightShift` + стрелки.

- `Ctrl+RightShift+Up`:
  - tap: play/pause
- `Ctrl+RightShift+Down`:
  - tap: repeat toggle (циклически: off → repeat_all → repeat_one → off, либо иной цикл — зафиксировать в реализации)
- `Ctrl+RightShift+Right`:
  - tap: next track
  - hold: fast-forward (seek +5s повтором)
- `Ctrl+RightShift+Left`:
  - tap: previous track
  - hold: rewind (seek -5s повтором)

Shuffle:
- Требование пользователя: “вверх+вниз — шафл”.
- Ограничение: большинство систем API глобальных хоткеев (в т.ч. Windows RegisterHotKey) плохо поддерживают “две обычные клавиши одновременно” как один хоткей (типа Up+Down) без модификаторов.
- Поэтому в конфиге должно быть предусмотрено:
  - либо отдельная комбинация для shuffle (рекомендуемый дефолт: `Ctrl+RightShift+S`),
  - либо экспериментальный режим “combo-detect”: считать shuffle включенным, если Up и Down нажаты в пределах окна 150–250ms (не гарантируется во всех случаях).

### Tap vs hold

- Должна быть логика распознавания tap/hold для Left/Right:
  - tap: одиночное действие (next/prev)
  - hold: периодическое действие (seek) пока удерживается
- Тайминги должны быть конфигурируемыми:
  - `hold_threshold_ms` (например 250–350ms)
  - `repeat_interval_ms` (например 200–300ms)

## Интерфейс (TUI/CLUI)

### Основной экран (до старта воспроизведения)

Должен отображать:
- активный плейлист
- список папок активного плейлиста
- подсказку меню:
  - `1` добавить папку
  - `2` удалить папку(и)
  - `3` воспроизвести выбранное
  - `4` настройки
  - `5` плейлисты

### Подменю “настройки”

Минимум:
- `min_size_bytes` (ввод числа, или выбор пресетов)
- `shuffle_mode` (on/off)
- `repeat_mode` (off/one/all)

### Подменю “плейлисты”

Минимум:
- список плейлистов
- создать новый
- удалить
- переименовать
- сделать активным (swap)

### Экран во время воспроизведения

Минимум:
- текущий трек (имя файла, можно без метаданных)
- номер в очереди / размер очереди
- статусы shuffle/repeat
- краткая подсказка хоткеев

## Нефункциональные требования

- **Легковесность**: минимальные зависимости, предпочтительно один бинарник.
- **Стабильность**: отсутствие падений при недоступных папках/файлах, корректные сообщения об ошибках.
- **Производительность**:
  - индексация не должна “вешать” UI надолго (желательно выполнять индексатор с прогрессом/в фоне).
- **Безопасность**:
  - не исполнять ничего из папок игр, только читать файлы,
  - аккуратно работать с путями и правами.

## Рекомендованный технологический стек (для агентов-исполнителей)

### Язык
- **Rust** (рекомендовано).

### Библиотеки (ориентир)

- TUI: `ratatui`, `crossterm`
- Конфиг: `serde` + `toml` (или `serde_json`)
- Файловый обход: `walkdir`
- Аудио без VLC: `symphonia` (decode) + `cpal` (output)
- Глобальные хоткеи (Windows): WinAPI `RegisterHotKey` через `windows` crate
- Linux (best effort): X11 grab keys (если будет реализовываться)

### Почему без VLC

- Требование: максимальная легковесность и portable-папка без тяжелых внешних runtime.
- VLC/libVLC усложняет упаковку и увеличивает размер дистрибутива. Можно рассмотреть как опциональный бэкенд позже, если возникнут проблемы с декодированием/seek.

## Критерии готовности (Acceptance Criteria)

На Windows 11:
- Приложение запускается из папки без установки.
- Создается `data/config.*` в папке приложения (или читается существующий).
- В меню можно:
  - добавить папку,
  - удалить папку,
  - создать/переключить плейлист,
  - поменять `min_size_bytes`, shuffle, repeat.
- После “воспроизвести” звук идет, даже если терминал свернут.
- Глобальные хоткеи работают:
  - play/pause
  - next/prev
  - FF/RW по удержанию (ступень 5с повтором)
  - repeat toggle
  - shuffle toggle (через конфиг/дефолтную альтернативную комбинацию)
- Файлы меньше порога **не попадают** в очередь.

## Открытые решения (зафиксировать в реализации)

- Конкретный формат конфигурации: TOML или JSON (оба подходят; TOML удобен для ручного редактирования).
- Как сортировать треки при построении очереди (по имени/пути/дате).
- Поведение “previous track”: вернуть на начало трека или перейти к предыдущему (зависит от позиции > N секунд).
- Реализация shuffle hotkey при требовании “Up+Down”: отдельная комбинация по умолчанию или эвристика.
