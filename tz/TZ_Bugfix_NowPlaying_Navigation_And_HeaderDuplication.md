## Цель

Устранить регрессии после внедрения пунктов ТЗ №2 (возврат/переход на экран Now Playing):

1) **Переход на Now Playing при старте воспроизведения** снова должен работать (не зависать на главном меню во время скана/запуска).
2) **Возврат на Now Playing для уже воспроизводимого плейлиста** должен работать надёжно.
3) Убрать **дублирование заголовка** “Now Playing” (как на скриншоте): оставить только заголовок блока/окна, без первой строки внутри страницы.

## Симптомы (как сейчас)

- Пункт 2 (“если выйти из Now Playing, вернуться без перезапуска”) **не срабатывает**.
- При запуске воспроизведения плейлиста перестал происходить переход на `NowPlaying` (пользователь остаётся в главном меню).
- На экране Now Playing видно дублирование:
  - заголовок блока/окна `Now Playing`
  - и первая строка текста внутри страницы тоже `Now Playing`

## Причины (по текущему коду)

- Дублирование заголовка вызвано тем, что `draw_now_playing()` рендерит блок с title `"Now Playing"` и одновременно кладёт строку `"Now Playing"` первой строкой контента.
  - Файл: `app/src/tui/ui.rs`, функция `draw_now_playing()`.
- Навигация при Play сейчас происходит **после завершения скана** (queue строится после скана), и экран может оставаться `MainMenu` до окончания скана.
  - Файл: `app/src/tui/app.rs`, ветка `Action::PlayerLoadFromLibrary { .. }` и `apply_scan_result(ScanOrigin::PlayFromLibrary { .. })`.
- “Уже играющий плейлист” определяется guard-условием через `active playlist id` + сравнение папок. Это может не сработать, если:
  - активный плейлист не установлен (например, воспроизведение было запущено “не из плейлиста”);
  - сравнение папок слишком строгое (учитывает поля, не важные для идентичности очереди) и даёт false negative.

## Требования к исправлению

### A) Убрать дублирование “Now Playing”

- Внутри страницы Now Playing **не должно быть** первой строки `"Now Playing"`.
- Остаётся только заголовок блока `title("Now Playing")`.

### B) Переход на Now Playing при старте воспроизведения

- При нажатии Play в главном меню:
  - экран должен **сразу переключаться на Now Playing** (оптимистично),
  - даже если сканирование/подготовка очереди ещё идёт.
- Пока скан идёт, в Now Playing допускается показывать:
  - `Status: stopped/paused` как есть
  - `Last error` / `status bar` со строкой вроде `"Scanning... (play pending)"` или `"Scanning..."`.
- Как только скан завершится и очередь загрузится — Now Playing остаётся активным (ничего дополнительно не нужно).

### C) “Если плейлист уже воспроизводится” → Navigate to Now Playing

- Если пользователь выбирает плейлист, который соответствует **уже воспроизводимому** набору треков, то:
  - не останавливать воспроизведение,
  - не пересканировать,
  - просто `Navigate(Screen::NowPlaying)`.

Чтобы это работало надёжно, нужно фиксировать **идентичность источника воспроизведения**:

- При `LoadQueue` (когда реально запускается воспроизведение очереди) сохранить в `AppState`/`TuiApp.state` поле вида:
  - `playback_source: PlaybackSource`
  - варианты: `FromActivePlaylist { id }` или `FromFoldersHash { hash }`
- При выборе плейлиста в меню сравнивать выбранный плейлист с `playback_source`:
  - если `FromActivePlaylist { id }` и `id` совпал — Navigate.
  - если источником был хэш папок — сравнивать хэш (по путям, без несущественных полей).

## Точки изменений (файлы)

- Переход при Play:

  - `app/src/tui/app.rs` → ветка `Action::LoadPlaylist { idx }`

  - `app/src/tui/app.rs` → место, где вызывается `PlayerCommand::LoadQueue` (`load_queue_from_current_library`) — записывать `playback_source`

## Acceptance Criteria

- На экране Now Playing отображается **только один** заголовок “Now Playing” (в title блока), внутри текста первой строки “Now Playing” нет.
- Нажатие Play в главном меню **сразу** переводит на Now Playing, даже если скан ещё выполняется.
- Если воспроизведение уже идёт и пользователь выбирает тот же плейлист — происходит **переключение на Now Playing** без Stop/LoadQueue/Rescan.
