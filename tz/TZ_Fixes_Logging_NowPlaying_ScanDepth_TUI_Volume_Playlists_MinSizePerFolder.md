## Цель

Реализовать пакет фиксов и улучшений для `ost_player`:

- управляемое логирование (уровни/фильтры) + ротация + ретеншен
- исправление навигации на экран Now Playing
- новый режим глубины сканирования “1 уровень”
- периодическое обновление TUI (1с/5с/не обновлять)
- дискретные уровни громкости (тонкая настройка на низких значениях)
- убрать лишнюю папку `data/playlists/`
- per-folder кастомный `min_size` (настройка + отображение + применение при скане)

## Контекст (где лежит)

- Конфиг: `data/config.yaml` (YAML), схема в `app/src/config/mod.rs`, загрузка/сейв в `app/src/config/io.rs`
- Логи: `data/logs/` (сейчас один файл `ost_player.log`), инициализация `app/src/logging.rs`
- Плейлисты: `data/playlists.yaml` (один файл), I/O `app/src/playlists/io.rs`
- TUI: `app/src/tui/*` (главный apply/reducer: `app/src/tui/app.rs`)
- Индексатор: `app/src/indexer/scan.rs`
- Плеер/снапшоты: `app/src/player/mod.rs`

## 1) Логирование: уровни + фильтры + ротация + ретеншен

### Требования

- Уровень логирования должен быть в конфиге (дефолт: **DEFAULT**).
- По умолчанию в лог пишем:
  - **все ERROR и FATAL**
  - **доменные события, меняющие конфигурацию**: создание/изменение/удаление плейлистов, изменения настроек
- События воспроизведения **не логировать** на дефолте.
- Шумные `WARN` от зависимостей (например `symphonia_bundle_mp3::demuxer: invalid mpeg audio header`) не должны попадать в лог по умолчанию.
- Логи должны быть разбиты на файлы: при дефолтном уровне логирования — **3 файла на месяц**.
- Логи старше **1 месяца** удаляются.
- В конфиге в секции комментариев должна быть справка по доступным уровням логирования.

### Предлагаемая реализация

- Добавить секцию `logging:` в `AppConfig`:
  - `default_level: Default|Debug|Trace` (или `Default|Verbose|Trace` — финальные имена зафиксировать)
  - `retention_days: 31` (или эквивалент “1 месяц”)
- `logging::init()` строит `EnvFilter`:
  - DEFAULT: `ost_player=info` (доменные инфо-события) и глобально `error`
  - Явно задушить зависимости до `error` (минимум: `symphonia*`, `rodio`)
  - Опционально: `RUST_LOG` может оверрайдить конфиг (или наоборот) — поведение фиксируется и описывается в комментариях конфига
- Ротация 3/месяц:
  - Нейминг по интервалам ~10 дней (например `YYYY-MM-01_10.log`, `YYYY-MM-11_20.log`, `YYYY-MM-21_eom.log`)
  - Writer выбирает имя файла по текущей дате и переоткрывает при смене интервала
- Ретеншен:
  - На старте (после инициализации логов или внутри неё) чистить `data/logs/` по `modified()` — удалять старше 31 дня

## 2) Now Playing: возврат на экран без перезапуска

### Требование

Если пользователь запускает воспроизведение плейлиста, который **уже воспроизводится**, приложение должно **просто перейти на экран Now Playing**, не перезапуская очередь.

### Точка внедрения

- `app/src/tui/app.rs`, обработка `Action::LoadPlaylist { idx }`:
  - ранний guard: если `player.status != Stopped` и выбранный плейлист совпадает с `state.playlists.active` (и/или совпадают `folders`) — делаем `Navigate(Screen::NowPlaying)` и выходим

## 3) Сканирование: новый режим “на один уровень вложенности” + значок `>⋮|`

### Требование

Добавить третий режим:

- сканировать выбранную папку и все её подпапки
- но **не** подпапки этих подпапок (depth = 1)

UI-иконки:

- `>>>` — рекурсивно (как сейчас)
- `>|⋮` — только корень (как сейчас)
- `>⋮|` — “на один уровень” (новый)

### Предлагаемая реализация

- Заменить bool `root_only` в `FolderEntry` на enum depth (3 состояния) с обратной совместимостью по YAML.
- Исправить сохранение выбранного режима глубины:
  - уровень глубины **должен сохраняться** при перезагрузке плейлистов
  - в файле плейлистов (`playlists.yml` / `playlists.yaml`) для каждой папки должен храниться маркер глубины сканирования **только если значение не дефолтное**
  - это обеспечивает преемственность со старыми файлами плейлистов; при сохранении новых данных файл может переписываться и “легаси-форма” исчезнет — это нормально
- В индексаторе (`app/src/indexer/scan.rs`) заменить флаг `recurse: bool` на ограничение глубины (0/1/∞) и параметр `cur_depth`.
- В UI (`app/src/tui/scan_indicator.rs`) маппить depth → нужная иконка.
- В меню заменить “toggle” на “cycle” по 3 состояниям (биндинг остаётся на той же клавише).

## 4) Обновление TUI по таймеру (вместо “только при действиях”)

### Требование

- Если окно **в фокусе и играет** → обновлять раз в **1 секунду**
- Если **играет xor в фокусе** → раз в **5 секунд**
- Если **свёрнуто** → можно **не обновлять**

### Предлагаемая реализация

- Текущее время в Now Playing зависит от `PlayerSnapshot.track_position`, а снапшоты сейчас почти не шлются.
- Поэтому нужно:
  - эмитить `PlayerSnapshot` периодически в `player::playback_thread()` (в `app/src/player/mod.rs`)
  - передавать в плеер состояние UI (focused/minimized) через новый `PlayerCommand::SetUiActivity { .. }`
  - состояние фокуса ловить в `app/src/tui/terminal.rs` через события `FocusGained/FocusLost` (crossterm), “minimized” — best effort (если недоступно, допускается fallback без “minimized”, но предпочтительно реализовать)

## 8) Подсказки по клавишам в UI должны подтягиваться из конфига

### Требование

- Подсказки по управлению, отображаемые в интерфейсе (блоки вида `Keys:` / список хоткеев на Now Playing), не должны быть захардкожены.
- Для **глобальных хоткеев** подсказки должны строиться из `config.yaml` → `hotkeys.bindings` (play/pause, next/prev, shuffle, repeat, volume).
- При изменении биндингов в конфиге пользователь должен видеть в UI актуальные комбинации.
- Дополнительно: **цифровые пункты меню** (действия в Main Menu) должны быть **конфигурируемыми**:
  - текущий порядок/назначения — порядок по умолчанию
  - если в конфиге задан кастомный порядок/назначения, то в UI пункты выводятся **в числовом порядке** (1..9), независимо от порядка в YAML

### Примечание

- Локальные TUI-клавиши (перемещение по меню, Enter/Esc в диалогах) можно оставлять статичными, т.к. они не настраиваются через `hotkeys.*`.

## 5) Дискретные уровни громкости

### Требование

Тонкая настройка на малой громкости:

- уровни: `0 1 2 3 5 7 10 13 16 20` и далее шаг 5%: `25 30 ... 100`
- в конфиге вместо `audio.volume_step_percent: 5` сделать `audio.volume_available_percent: [ ... ]`
- `audio.default_volume_percent` переименовать в `audio.volume_default_percent`

### Предлагаемая реализация

- В конфиге:
  - `audio.volume_available_percent: Vec<u8>` (валидация: уникальный, сортированный, 0..=100, содержит 0 и 100)
  - миграция: принимать старое `volume_step_percent` и при отсутствии списка генерировать `[0, step, 2*step, ... 100]`
  - алиас/миграция для `default_volume_percent` → `volume_default_percent`
  - легаси совместимость для `config.yml` / `config.yaml` **не требуется** (конфиг может быть создан заново)
- В TUI:
  - `VolumeUp/Down` вычисляют следующий/предыдущий уровень из списка (а не ±step) и шлют в плеер `SetVolumePercent(next)`

## 7) Кастомный `min_size` для каждой папки

### Требования

- Ввести поддержку кастомного `min_size` для каждой папки (override глобального порога).
- Отображать значение в списке папок активного плейлиста:
  - после знака глубины сканирования и перед путём до папки
  - символ:
    - `◼` (U+25FC) — используется дефолтный (глобальный) `min_size`
    - `🄲` (U+1F132) — используется кастомный `min_size`
- В меню активного плейлиста добавить команду **4**: “задать кастомный min_size”.
- В текущем меню активного плейлиста пункт **Play** сейчас соответствует `4` / `Enter` / `Space`. Требуется сдвиг нумерации:
  - **Play переезжает на 5**
  - все пункты ниже (до `Space`) сдвигаются на **+1**
  - `0` / `q` (exit) остаётся без изменений
- Доп. правки текста/вёрстки меню (чтобы Actions помещался по ширине и был компактнее):
  - сократить названия пунктов:
    - `cycle scan depth for selected folder` → `folder scan depth`
    - `set custom min_size for selected folder` → `folder min_size`
    - `remove selected folder` → `remove folder`
  - убрать `5` и `0` из цифрового блока действий, оставить отдельным блоком после пустой строки:\n    `Enter / Space  play`\n    `Esc / q       exit`
- Значение задаётся/отображается в **килобайтах**.
- Диапазон: **10..=10000 KB**.
  - всё вне диапазона **игнорируется**
  - сам диапазон **прописан в конфиге**
- При выборе папки в UI показывать её эффективный `min_size` в статус-строке (как сейчас для глобального min_size). Для папок без кастома показывать величину стандартного значения.
- Кастомный `min_size` должен реально применяться при сканировании/индексации (фильтрация по размеру per-folder).

### Предлагаемая реализация

- Конфиг:
  - `FolderEntry.min_size_kb: Option<u32>` (или `Option<u16>` при достаточном диапазоне), `None` = использовать глобальный `settings.min_size_bytes`.
  - В `settings` добавить:
    - `min_size_custom_kb_min: 10`
    - `min_size_custom_kb_max: 10000`
  - Валидация/нормализация:
    - если `Some(v)` и `v` вне диапазона → сбрасывать в `None` (игнорировать)
- UI:
  - В `app/src/tui/ui.rs` при выводе строки папки показывать `◼`/`🄲` и эффективное значение (KB).
  - Команда “4” в меню активного плейлиста:
    - UI ввод числа (KB), проверка диапазона; вне диапазона — статус “ignored”
    - сохранять в конфиг и сразу отражать в списке/статусе
  - При навигации по списку папок обновлять status-строку: показывать эффективный `min_size` выбранной папки.
- Индексатор:
  - Протащить effective `min_size_bytes` до `scan_dir` (на уровне `FolderScanEntry` либо как часть `ScanOptions`) и фильтровать файлы по этому значению именно для соответствующего folder root.

## Acceptance Criteria (проверки)

- DEFAULT логирование:
  - нет `WARN` от зависимостей в логе
  - есть ERROR/FATAL
  - есть записи о создании/изменении плейлистов и настроек
- В `data/logs/` создаются файлы по схеме 3/месяц; старше 31 дня удаляются при запуске.
- Экран Now Playing: при выборе уже играющего плейлиста — просто переход на Now Playing без перезапуска.
- Скан depth=1 работает (корень + подпапки на 1 уровень).
- Таймер Now Playing обновляется без действий: 1с/5с/0с по политике.
- Громкость меняется по заданным дискретным уровням.
- `data/playlists/` не создаётся.
- Per-folder min_size:
  - символ `◼`/`🄲` отображается корректно
  - ввод вне диапазона игнорируется
  - effective min_size в статусе совпадает с отображением
  - фильтрация реально меняется per-folder.
