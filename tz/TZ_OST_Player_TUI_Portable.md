# Техническое задание (ТЗ): локальный OST‑плеер из папок (portable, TUI, global hotkeys)

## 0) Контекст и цель
Разработать локальный плеер для OST/музыки из папок игр. Источники музыки — папки на диске (например Steam library, `Games/…`), где лежат треки в открытых форматах (`.mp3`, `.ogg`). Приложение предназначено для фонового воспроизведения во время работы/игры и должно управляться **глобальными** горячими клавишами, чтобы не отвлекаться на окно плеера.

## 1) Платформы и режим поставки
- **Основная платформа**: Windows 11.
- **Linux**: поддержку можно **исключить из MVP**, но кодовую базу нужно писать так, чтобы добавление Linux в будущем не требовало переписывания “с нуля” (см. раздел 12).
- **Portable‑режим обязателен**:
  - приложение не устанавливается в систему;
  - **все файлы приложения** (бинарник + настройки + плейлисты + кэш) находятся **в одной папке**, расположенной в произвольном месте;
  - приложение **не** пишет в `%APPDATA%`, реестр, `~/.config`, systemd и т.п.
- **Права на запись**:
  - если папка приложения недоступна на запись (например, находится в `Program Files`), приложение должно:
    - работать в режиме “только чтение” для настроек недопустимо;
    - показать понятную ошибку, что требуется перенести папку приложения в место с правами записи (например `D:\Apps\ost_player\`), и завершиться.

## 2) Пользовательский сценарий (workflow)
### 2.1 Старт
Пользователь запускает приложение из терминала (PowerShell/CMD/Windows Terminal).

После запуска приложение показывает TUI‑интерфейс со списком добавленных папок и меню действий.

### 2.2 Подготовка источников
Пока воспроизведение не запущено, приложение должно циклично оставаться в меню, позволяя:
- добавлять папки,
- удалять папки,
- менять настройки,
- управлять плейлистами (см. раздел 5).

### 2.3 Запуск воспроизведения и фон
Пользователь выбирает “воспроизвести”.
После старта музыка должна продолжать играть, даже если окно терминала **свернуто**.

## 3) Интерфейс (TUI / CLUI)
### 3.1 Общие требования
- Интерфейс — текстовый (ASCII/Unicode), работающий в терминале.
- Управление в меню — цифрами/клавишами (не мышью).
- Отображать понятные статусы (сколько папок, сколько найдено треков, текущие настройки).

### 3.2 Главное меню (до старта воспроизведения)
После запуска отобразить:
- **Список добавленных папок** (с индексами).
- Меню:
  1. **Добавить папку**
  2. **Удалить папку(и)**
  3. **Воспроизвести выбранное**
  4. **Настройки** (`size threshold`, `shuffle mode`, `repeat mode`)
  5. **Плейлисты** (см. раздел 5) — отдельный пункт, интегрированный в меню
  0. Выход

Примечание: точные названия пунктов могут быть адаптированы, но логика должна соответствовать.

### 3.3 Экран настроек
Настройки, доступные из меню:
- `min_size_bytes` (минимальный размер файла для добавления в плейлист)
- `shuffle` (вкл/выкл)
- `repeat` (off / one / all) — минимально достаточно `off` и `one/all` по выбору реализации

### 3.4 Экран “Now playing” (после старта воспроизведения)
Минимально отображать:
- текущий трек (имя файла; по возможности — путь)
- режимы: shuffle/repeat
- прогресс (если доступен)
- подсказку по основным хоткеям

## 4) Источники треков и фильтрация
### 4.1 Источники
- Пользователь добавляет **папки** (директории) как источники.
- Воспроизведение строится из содержимого этих папок (рекурсивно).

### 4.2 Поддерживаемые форматы (MVP)
- `.mp3`
- `.ogg`

Расширение списка форматов — опционально и не должно ломать MVP.

### 4.3 Исключение мусора по размеру файла
Плеер **не должен** добавлять в плейлист аудиофайлы меньше заданного порога:
- критерий: `file_size_bytes >= min_size_bytes`
- размер берётся из метаданных файловой системы (не из декодера)

### 4.4 Поведение при ошибках чтения/доступа
- Если часть папок/файлов недоступна (нет прав/удалено/битые пути), приложение должно:
  - пропустить недоступные элементы,
  - показать пользователю счётчик ошибок/предупреждение,
  - не падать целиком.

## 5) Плейлисты (сохранение и swap)
### 5.1 Определение плейлиста
**Плейлист** — это именованный блок в **отдельном файле плейлистов**:
- `name`: строка
- `folders`: массив путей к папкам (источникам)

Плейлист хранит **только** набор папок. Настройки воспроизведения (min size, shuffle, repeat, hotkeys) — глобальные (если не оговорено иначе).

### 5.2 Операции над плейлистами (через меню “Плейлисты”)
Должны быть доступны:
- просмотр списка плейлистов
- создать новый плейлист (с вводом имени, из текущего списка папок)
- сохранить текущие папки в существующий плейлист (overwrite/подтверждение)
- загрузить плейлист (swap): текущий список папок заменяется на `folders` выбранного плейлиста
- переименовать плейлист
- удалить плейлист (с подтверждением)

### 5.3 Поведение при swap
- После загрузки плейлиста приложение возвращается в главное меню (до воспроизведения).
- Если воспроизведение уже идёт, поведение swap должно быть определено одним из вариантов (выбрать в реализации и явно задокументировать):
  - A: остановить воспроизведение, загрузить папки, перейти в меню
  - B: применить новый плейлист как новую очередь со следующего трека

Для MVP допускается вариант A (проще и предсказуемее).

## 6) Конфигурация (файл настроек)
### 6.1 Расположение и структура папок приложения
В папке приложения:
- `./ost_player.exe` (или иной бинарник)
- `./data/` (обязательная папка данных)
  - `config.yaml` (единый файл конфигурации)
  - `playlists.yaml` (отдельный файл плейлистов: Current + именованные)
  - (опционально) `cache/`

### 6.2 Требования к формату
- Конфигурация должна быть **человекочитаемой** и редактируемой вручную.
- **Формат**: YAML.
- Конфиг должен включать:
  - список текущих папок (активный набор)
  - `min_size_bytes`
  - `shuffle`
  - `repeat`
  - `hotkeys` (см. раздел 7)
  - плейлисты в конфиг **не** включать (они живут в `./data/playlists.yaml`)

### 6.3 Автосохранение
- Любое изменение папок/настроек через меню должно сохраняться на диск в `./data/config.yaml` без ручного “Save”.
- Любое изменение плейлистов должно сохраняться на диск в `./data/playlists.yaml` без ручного “Save”.
- При старте приложение загружает `./data/config.yaml` и `./data/playlists.yaml`. Если файлов нет — создаёт дефолтные.

## 7) Глобальные горячие клавиши (Windows 11 — обязательно)
### 7.1 Общие требования
- Хоткеи должны работать **глобально**: вне фокуса терминала, пока приложение работает.
- Хоткеи должны настраиваться пользователем через конфиг.

### 7.2 Дефолтные биндинги (обязательно)
По умолчанию:
- модификаторы: `Ctrl + RightShift`
- клавиши: стрелки

Семантика:
- `Ctrl+RightShift+Up`: **Play/Pause** (короткое нажатие)
- `Ctrl+RightShift+Down`: **Repeat toggle** (короткое нажатие)
- `Ctrl+RightShift+Left`:
  - короткое нажатие: **Previous track**
  - удержание: **Rewind** (fast backward)
- `Ctrl+RightShift+Right`:
  - короткое нажатие: **Next track**
  - удержание: **Fast forward**
- `Ctrl+RightShift+Up` + `Ctrl+RightShift+Down` (“вверх+вниз”): **Shuffle toggle**
  - Важно: одновременный хоткей на две стрелки может быть сложен/ненадёжен. Требование: *shuffle должен быть доступен по хоткею*. Реализация может выбрать более надёжную дефолтную комбинацию (например `Ctrl+RightShift+S`), но тогда нужно:
    - оставить возможность настроить желаемое поведение через конфиг,
    - документировать выбранный дефолт.

### 7.3 Tap vs hold
- Для `Left/Right` поддерживать различение:
  - **tap**: переключение трека
  - **hold**: перемотка
- Порог “tap vs hold” должен быть параметром (дефолт можно зафиксировать), но как минимум должен существовать разумный дефолт (например 250–350 мс).

### 7.4 Поведение перемотки (дефолт)
По умолчанию перемотка при удержании:
- “ступеньками” по **5 секунд** с повтором (например каждые 200–300 мс повторять команду seek на 5 секунд).

### 7.5 Конфликты хоткеев
- Если зарегистрировать глобальный хоткей не удалось (занят другой программой), приложение должно:
  - показать понятное сообщение (какой хоткей не зарегистрирован),
  - предложить изменить конфиг,
  - продолжить работу (без этого хоткея), либо завершиться — выбрать и задокументировать (для MVP предпочтительнее продолжить).

## 8) Логика воспроизведения
### 8.1 Очередь (queue)
- При старте воспроизведения строить список треков из активного набора папок.
- Порядок:
  - при `shuffle=off`: детерминированный (например по пути/имени файла) — чтобы поведение было стабильным
  - при `shuffle=on`: перемешивание очереди

### 8.2 Repeat
Поддержать:
- `repeat=off`: дойдя до конца — остановить
- `repeat=all`: по окончании — начать сначала
- (опционально) `repeat=one`: зациклить текущий трек

### 8.3 Обработка ошибок декодирования
- Если файл не декодируется — пропустить и перейти к следующему, зафиксировав ошибку в статусе/логе.

## 9) Нефункциональные требования
- **Лёгкость**: избегать тяжёлых рантаймов; нежелательно тянуть VLC/libVLC в MVP.
- **Стабильность**: приложение не должно падать от битых файлов/папок.
- **Предсказуемость**: понятные сообщения об ошибках (нет прав на запись, папка не найдена, хоткей занят).

## 10) Рекомендованный стек (для агентов-исполнителей)
Для Windows‑first portable TUI:
- Rust
- TUI: `ratatui` + `crossterm`
- Конфиг: `serde` + YAML
- Сканирование: `walkdir`
- Аудио: `symphonia` (decode mp3/ogg) + `cpal` (output)
- Хоткеи Windows: WinAPI `RegisterHotKey` (через crate `windows`)

Linux — позже/опционально; если делать:
- хоткеи: X11 grab keys (X11) или альтернативные механизмы под Wayland/DE (может потребовать отдельной интеграции)

## 11) Критерии готовности (Definition of Done для MVP)
- Приложение запускается в терминале и показывает меню со списком папок.
- Можно добавить/удалить папки, изменения сохраняются в `./data/config.yaml`.
- Есть меню настроек: `min_size_bytes`, `shuffle`, `repeat`.
- Есть меню “Плейлисты”: создать/удалить/переименовать/загрузить(swap) плейлист (плейлист = имя + папки).
- Воспроизведение mp3/ogg работает; при `min_size_bytes` маленькие файлы не попадают в очередь.
- После старта воспроизведения музыка играет при свернутом терминале.
- Глобальные хоткеи на Windows работают по дефолту и настраиваются через конфиг; `tap vs hold` для Left/Right; перемотка по 5с при удержании.

## 12) Linux в будущем: ожидаемые сложности и требования к архитектуре (чтобы не переписывать)
### 12.1 Основные сложности интеграции на Linux
- **Глобальные хоткеи**:
  - на **X11**: относительно прямолинейно (grab keys).
  - на **Wayland**: “любые физические клавиши глобально” часто ограничены политиками безопасности/DE; могут понадобиться порталы, MPRIS, или интеграция с конкретным окружением.
- **Аудио**:
  - вывод звука кроссплатформенно решаем, но на Linux нужно корректно работать с PulseAudio/PipeWire/ALSA через выбранный backend.
- **Упаковка**:
  - portable на Linux обычно означает “папка с бинарником и конфигами”, но зависимости/устройства могут отличаться; важно избегать системных установок по требованию пользователя.

### 12.2 Требования к проектированию (обязательны уже в Windows‑версии)
Чтобы позже добавить Linux без переписывания:
- **Абстракции по платформам**:
  - выделить интерфейс `HotkeyProvider` (Windows/X11/Wayland‑future).
  - выделить интерфейс `AudioBackend` (как минимум один backend, но API должен не быть завязан на WinAPI).
- **Изоляция WinAPI**:
  - весь WinAPI‑код должен быть в отдельном модуле/крейте, без “протекания” типов WinAPI в остальной код.
- **TUI не зависит от платформы**:
  - TUI и core‑логика (индексатор, очередь, shuffle/repeat, хранение YAML) должны быть максимально platform‑agnostic.
- **Событийная модель**:
  - глобальные хоткеи и ввод в TUI сводятся к общим командам (`PlayerCommand`), чтобы подключение нового провайдера хоткеев не затрагивало аудиоплеер и меню.
