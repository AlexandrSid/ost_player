# ТЗ: UI-фиксы — символы сканирования + нумерация меню + TextInput (Add folder)

## 0) Контекст

В текущей сборке TUI:

- В списке папок/настройках режима сканирования используются **плохо читаемые** и местами **перепутанные** символы “рекурсия vs только корень”.
- В `Main menu` пункты отображаются **неконсистентно**: часть имеет формат `N / key`, а пункт `toggle root_only...` показывает только `t` без цифровой кнопки.
- В диалоге `Add folder (absolute path)` поле ввода:
  - визуально “узкое” по высоте (псевдополе),
  - **не отображает вводимый/редактируемый текст** (виден только курсор/каретка).

Это ТЗ предназначено для реализации трёх UX-улучшений.

## 1) Цели

- Сделать индикаторы режимов сканирования **однозначными и читаемыми**.
- Привести главное меню к **строго пронумерованному** виду (каждый пункт начинается с цифры) и добавить **цифровую кнопку** для `toggle root_only`.
- Исправить виджет ввода пути при добавлении папки: **высота 1 строка**, **видимый текст**, **корректный курсор**.

## 2) Область изменений (ориентиры по текущему коду)

Ожидаемые точки:

- Главное меню: `app/src/tui/screens/main_menu.rs` (рендер текста “Actions / Main menu”, биндинги клавиш).
- Виджет ввода: `crate::tui::widgets::TextInput` (используется в `main_menu.rs` для `Add folder`).
- Отображение списка папок и статуса сканирования: `app/src/tui/ui.rs` (или соседние модули UI, где формируется строка для каждой папки).

## 3) Требование A: символы режимов сканирования (root-only vs recursive)

### 3.1 Новые символы

- **Рекурсивное сканирование**: показывать как **`>>>`**
- **Сканирование только корня**: показывать как **`>|⋮`**

Важно: символы должны быть **визуально моноширинными** в контексте UI (колонки не “пляшут”, строки остаются выровненными).

### 3.2 Правильная семантика

- `>>>` означает **recursive** (обход подпапок включён).
- `>|⋮` означает **root-only** (сканировать только файлы в корне выбранной папки).

### 3.3 Критерии готовности (символы)

- Везде, где показывается режим сканирования, символы **не перепутаны** и соответствуют 3.2.
- Строки UI остаются читаемыми и выровненными: индикатор занимает фиксированное место (при необходимости — паддингом/выравниванием).

## 4) Требование B: главное меню — консистентная нумерация + цифровая кнопка на toggle root_only

### 4.1 Отображение

- **Каждый** пункт главного меню должен **начинаться с цифры** (единый формат).
- Разрешено оставлять “буквенные” хоткеи и Enter/Space как вторичные подсказки, но пункт всё равно должен иметь номер.
- Легаси-нумерацию и порядок соблюдать **не обязательно** — допускается перенумерация/переупорядочивание, если это улучшает консистентность.

Рекомендуемый формат строки (допускается эквивалент):

- `N / <key(s)>  <action>`

### 4.2 Биндинги (минимальный набор)

Нужно добавить цифровую кнопку для режима сканирования, который сейчас на `t`:

- `toggle root_only for selected folder` должен иметь:
  - **цифровой биндинг** (например `3`)
  - и **сохранить** букву `t` (как быстрый хоткей), если не конфликтует

Остальные пункты должны иметь цифровые биндинги и отображаться с номером.

### 4.3 Предлагаемая целевая раскладка (принять как стандарт, если нет возражений)

- `1 / a   add folder`
- `2 / d   remove selected folder`
- `3 / t   toggle root_only for selected folder`
- `4 / Enter / Space  play`
- `5 / s   settings`
- `6 / p   playlists`
- `7 / r   rescan library`
- `0 / q   exit`

Если текущая обработка уже использует эти цифры для других действий, допускается другая нумерация при условии выполнения 4.1–4.2.

### 4.4 Критерии готовности (главное меню)

- В главном меню **нет** пунктов без цифры.
- `toggle root_only...` вызывается как минимум:
  - по **цифре** (новый биндинг),
  - по `t` (если сохранено).
- Отображение меню визуально консистентно (единый формат строк).

## 5) Требование C: Add folder — исправить TextInput (высота 1 строка + видимый текст)

### 5.1 Проблема

В окне `Add folder (absolute path)`:

- “поле ввода” визуально слишком узкое по высоте (псевдополе),
- вводимые/редактируемые символы **не отображаются** (виден только курсор).

### 5.2 Новое поведение (UI/UX)

- Поле ввода должно быть **высотой ровно 1 строка текста** (без “сплющивания”).
- Внутри поля ввода должны отображаться:
  - текущий текст,
  - ввод новых символов,
  - редактирование (Backspace/Delete),
  - перемещение курсора (Left/Right, Home/End — если поддерживается).
- Если текст длиннее ширины поля:
  - должен работать “видимый фрагмент” строки (scroll/viewport),
  - курсор **всегда** видим, даже при длинном пути.
- Курсор должен соответствовать позиции редактирования внутри отображаемого фрагмента.

### 5.3 Критерии готовности (TextInput)

- При наборе с клавиатуры текст **виден**.
- При вставке из буфера обмена текст **виден**.
- Backspace/Delete изменяют буфер и это видно.
- При длинном пути ввод/курсор не “пропадают”: видимая часть строки следует за курсором.
- Высота поля ввода визуально соответствует **1 строке**.

## 6) Нефункциональные требования

- Не ломать существующую навигацию (Up/Down, Enter, Esc).
- Поведение должно быть стабильным в Windows терминалах в рамках используемой TUI-библиотеки.

## 7) Мини-тест план

- **Символы режимов скана**:
  - переключить режим для выбранной папки → проверить, что `>>>` = recursive, `>|⋮` = root-only,
  - убедиться, что выравнивание строк не ломается.
- **Главное меню**:
  - проверить, что у каждого пункта есть цифра,
  - проверить, что `toggle root_only...` работает по цифре и по `t` (если оставили).
- **Add folder / TextInput**:
  - набрать короткий путь → текст отображается,
  - набрать длинный путь → курсор и ввод видимы (viewport/scroll),
  - проверить Backspace/Delete и перемещение курсора,
  - Enter = save, Esc = cancel.
