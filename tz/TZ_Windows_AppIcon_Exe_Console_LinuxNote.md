## ТЗ: внедрение иконки приложения (Windows .exe + иконка консоли) + заметка для Linux

## 0) Цель
- **Windows**: использовать `app/resources/app_icon.ico` как:
  - значок итогового `ost_player.exe` (Explorer/ярлыки/закрепление на панели задач),
  - **и** (best-effort) значок окна консоли во время работы приложения.
- **Linux**: если появится Linux-версия/упаковка — не пропустить требование про иконку (для Linux это решается на этапе desktop/packaging, а не “встраиванием в бинарник” обычной сборкой Cargo).

## 1) Входные данные
- **Файл иконки**: `app/resources/app_icon.ico`.
- **Целевой бинарник**: `ost_player.exe` (сборка `cargo build --release` из директории `app/`).

## 2) Требование A (Windows): иконка в ресурсах .exe
### 2.1 Ожидаемое поведение
- После сборки `app/target/release/ost_player.exe` должен иметь корректную иконку:
  - в Проводнике (крупные/мелкие значки),
  - у ярлыка,
  - при “Pin to taskbar/Start”.

### 2.2 Технический подход (рекомендуемый)
- Добавить в `app/` компиляцию Windows-ресурсов на этапе Cargo build:
  - вариант 1 (предпочтительно): `build.rs` + `winres` (как build-dependency),
  - вариант 2: `build.rs` + `embed-resource` / или ручной `.rc` + `rc.exe` (если `winres` окажется неудобным).

### 2.3 Требования к ICO (качество)
- Внутри `app_icon.ico` должны быть как минимум размеры:
  - 16×16, 32×32, 48×48, 256×256,
  - желательно также 24×24, 64×64.
- Если некоторых размеров нет и Windows показывает “мыло/плохую” иконку — это считается дефектом входного ассета и должно быть исправлено пересборкой ICO.

### 2.4 Критерии готовности (A)
- `ost_player.exe` отображает нужную иконку в Explorer (в нескольких размерах).
- При создании ярлыка/закреплении — иконка остаётся корректной.

## 3) Требование B (Windows): иконка окна консоли во время работы
### 3.1 Ожидаемое поведение
- При запуске приложения значок окна:
  - **в классическом ConHost** (cmd.exe/PowerShell без Windows Terminal) должен смениться на иконку приложения,
  - **в Windows Terminal** — best-effort (может игнорировать WM_SETICON; это не блокер).

### 3.2 Технический подход (WinAPI, best-effort)
- Под `cfg(windows)` добавить вызов, который:
  1) Получает HWND консольного окна через `GetConsoleWindow()`.
  2) Загружает `HICON`:
     - предпочтительно из ресурсов текущего модуля (`GetModuleHandleW(NULL)` + `LoadImageW/LoadIconW` + `MAKEINTRESOURCEW(<ID>)`), где `<ID>` соответствует icon resource, встроенному в `.exe`.
  3) Вызывает:
     - `SendMessageW(hwnd, WM_SETICON, ICON_SMALL, hicon)`
     - `SendMessageW(hwnd, WM_SETICON, ICON_BIG, hicon)`
- В случае любых ошибок:
  - не падать,
  - залогировать причину (debug/warn) и продолжить работу.

### 3.3 Важная деталь: идентификатор ресурса иконки
- В ТЗ фиксируется требование обеспечить **стабильный** идентификатор icon resource (например `1`), чтобы runtime-код мог загрузить иконку.
- Если выбранный инструмент (`winres` и т.п.) не гарантирует простой ID — допускается перейти на явный `.rc`, где задаётся `IDI_APPICON ICON "app/resources/app_icon.ico"`, и в runtime использовать этот ID.

### 3.4 Критерии готовности (B)
- В ConHost при запуске отображается иконка приложения.
- В Windows Terminal поведение задокументировано как best-effort (может не меняться, и это допустимо).

## 4) Linux: заметка для текущей build-инструкции и для будущей упаковки
### 4.1 Текущее состояние
- Сейчас сборка описана как `cargo build --release` (см. `[app/README.md](app/README.md)`).

### 4.2 Требование (чтобы не забыть при появлении Linux-версии)
- Для Linux **обычная** сборка Cargo (`cargo build --release`) не “встраивает” иконку в ELF так, чтобы она отображалась в меню приложений.
- Если появится Linux-дистрибутив/запуск из меню, необходимо добавить packaging-артефакты:
  - `.desktop` файл (Desktop Entry),
  - PNG-иконки разных размеров в `hicolor` (например 256/128/64/48/32/16),
  - шаг установки/упаковки (deb/rpm/AppImage/flatpak — по выбранному формату).
- В документации сборки Linux должен быть отдельный подпункт “Icons”, чтобы требование не потерялось.

## 5) Мини-тест-план
### Windows
- Собрать `cargo build --release` (в `app/`).
- Проверить:
  - иконку `ost_player.exe` в Explorer (List/Details + Large icons),
  - иконку ярлыка,
  - иконку после “Pin to taskbar”.
- Запустить:
  - из `cmd.exe`/PowerShell (ConHost) → проверить иконку окна,
  - из Windows Terminal → зафиксировать, меняется ли (best-effort).

### Linux
- Проверить, что в build-инструкции упомянуто: иконка — это desktop/packaging задача, а не часть `cargo build`.
